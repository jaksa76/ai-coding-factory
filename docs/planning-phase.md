# Planning Phase — Functional Requirements

## 1. Overview

Currently, when a worker claims a Jira issue it proceeds directly to implementation with no opportunity for a human to review the intended approach beforehand. This can lead to wasted effort if the agent's interpretation diverges from the stakeholder's intent.

The planning phase introduces an optional review gate: before writing any code the agent produces a written plan, publishes it to the target repository, and waits for a human to approve it. Only after approval does the agent begin implementation.

---

## 2. Opt-In / Opt-Out Mechanism

Whether planning is required is determined by two controls that interact:

### Project-wide default

The environment variable **`PLAN_BY_DEFAULT`** sets the default for the entire project:

| Value | Meaning |
|---|---|
| unset or `false` | Planning is **opt-in** (default behaviour) |
| `true` | Planning is **opt-out** — all issues go through the planning phase unless individually exempted |

### Per-issue override via Jira labels

A label on the Jira issue can override the project-wide default:

| Label | Effect |
|---|---|
| `needs-plan` | Force planning for this issue, even when `PLAN_BY_DEFAULT` is `false` |
| `skip-plan` | Skip planning for this issue, even when `PLAN_BY_DEFAULT` is `true` |

If both labels are present on the same issue, `skip-plan` takes precedence.

### Decision matrix

| `PLAN_BY_DEFAULT` | Issue label | Planning required? |
|---|---|---|
| `false` (or unset) | _(none)_ | No |
| `false` (or unset) | `needs-plan` | Yes |
| `false` (or unset) | `skip-plan` | No |
| `true` | _(none)_ | Yes |
| `true` | `needs-plan` | Yes |
| `true` | `skip-plan` | No |

---

## 3. Jira Workflow

### With `needs-plan`

```
To Do → Planning → Awaiting Plan Review → Plan Approved → In Progress → Done
```

| Transition | Meaning |
|---|---|
| To Do → Planning | Worker has claimed the issue and started work |
| Planning → Awaiting Plan Review | Plan has been published; human review is required |
| Awaiting Plan Review → Plan Approved | Human has approved the plan; agent proceeds to implementation |
| Plan Approved → In Progress | Implementation is underway |
| In Progress → Done | Implementation is complete |

### Without `needs-plan`

```
To Do → In Progress → Done
```

The existing behaviour is unchanged for unlabelled issues.

---

## 4. Plan Document

| Property | Value |
|---|---|
| Location | `plans/<ISSUE-KEY>.md` in the **target repository** (the repo being worked on, not `ai-coding-factory`) |
| Format | Markdown |
| Editability | A human reviewer may edit the file directly on GitHub before approving |

Plans are comitted to the main branch of the target repository.

The document contains the implementation plan generated by the agent for the given issue. It is committed and pushed to the target repository as part of the planning phase.

---

## 5. Jira Issue Linkage

After pushing the plan file, the worker adds a comment to the Jira issue containing:

- A direct link to the plan file on GitHub (e.g. the rendered view of `plans/<ISSUE-KEY>.md` on the default branch).
- It transitions the issue from `Planning` to `Awaiting Plan Review`.

This comment is the primary notification mechanism; no other alerting is provided.

---

## 6. Approval

- The reviewer reads the plan file on GitHub.
- The reviewer may edit the file directly in the GitHub UI before approving.
- Approval is signalled by transitioning the Jira issue from **`Awaiting Plan Review`** to **`Plan Approved`**.
- No other approval mechanism (pull request, comment keyword, etc.) is required.

---

## 7. Worker Roles

To maximise throughput and avoid workers blocking on human review, two distinct worker roles are defined:

### Planner worker

- Claims issues that require planning (per the rules in §2) from the `To Do` queue.
- Generates the plan, commits it to the target repository, posts the Jira comment, and transitions the issue to `Awaiting Plan Review`.
- Immediately moves on to the next available issue — it does **not** wait for approval.

### Implementer worker

- Claims work from two queues:
  1. `To Do` issues that do **not** require planning.
  2. `Plan Approved` issues (plan has been reviewed and approved).
- Implements the work using the existing flow and transitions to `Done`.
- Has no involvement in the planning phase.

### Separation of concerns

Neither worker type ever blocks waiting for a human. The approval window is dead time for the issue only, not for any running worker. Planner and implementer workers can be scaled independently based on the mix of work.

---

## 8. New Jira Statuses Required

Any Jira project that uses the planning phase must have the following two statuses configured:

| Status | Purpose |
|---|---|
| `Planning` | Indicates the agent is generating the plan |
| `Awaiting Plan Review` | Indicates the plan is ready for human review |
| `Plan Approved` | Indicates the plan has been approved and implementation can begin |

These statuses must be reachable via transitions from the standard statuses (`In Progress`, etc.) in the project's workflow.

### Behaviour when statuses are absent

If the worker cannot find or transition to the required statuses, it falls back to the non-planning flow and logs a warning. It does not fail or leave the issue in a broken state.

The warning is emitted in both cases below, but with different severity:

| Situation | Warning level |
|---|---|
| `PLAN_BY_DEFAULT=true`, no `needs-plan` label | Standard warning — project default cannot be honoured |
| `needs-plan` label explicitly set on the issue | Prominent warning — an explicit request is being silently bypassed |

In all cases the worker proceeds with implementation immediately rather than blocking.

---

## 9. Constraints and Non-Goals

- **Opt-in only.** Planning is not mandatory; issues without `needs-plan` are unaffected.
- **No implementation during planning.** The agent produces only the plan document; no code changes are made to the target repository until after approval.
- **Agent responsibility.** The worker does not enforce that the plan file is committed before transitioning to `Awaiting Plan Review`; correct behaviour is the agent's responsibility.
- **One optional environment variable.** `PLAN_BY_DEFAULT=true` enables project-wide planning; when unset the behaviour is opt-in per issue, matching the previous default.
- **Single approval path.** The only supported approval mechanism is a Jira status transition; no alternative paths are defined.
